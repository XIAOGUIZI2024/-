# 最左前缀匹配原则
使用联合索引时，查询条件必须从索引的最左边开始匹配，只有满足最左边的条件，才可以进行下一列的匹配，但是在Mysql 5.6 版本引入了索引下推的优化，如果优化器它可能会选择索引下推，就会导致最左前缀匹配原则不生效

# 什么是回表
在mysql存储引擎中二级索引的b+树结构中叶子节点存储的是完整数据的主键，如果插查询的数据没有在二级索引的叶节点上，要获取叶子节点上的主键去聚簇索引上查询获取完整的数据，这个就是回表

# mysql获取数据一定走索引吗
不一定的，这个主要是看mysql里面的优化器的选择，如果全表扫描在cup和IO等成本比较低，那可能会选择全表扫描，我们可以通过在执行的sql语句前添加explain要查看执行sql的效果，在type这个字段上all是全表扫描，range是范围扫描，index是索引扫描，ref是等值匹配
key 这个字段如果不是null 就是不走索引 如果有值的话，那就是走索引
rows 这个字段代表的是扫描的数据是多少

# 索引失效问题
* 没有符合最左匹配原则
* 索引上做运算
* 索引上使用了函数
* like左则使用了模糊查询
* or连接非索引字段
* 优化器评估成本没有选择走索引
# 索引建立的要求
* 选择建立索引的字段要有区分度
* 大字段不适建立索引
* 写多读少的表也不适合建立索引
* 频繁查询的字段要建立索引
*  排序 和 分类的字段要建立索引
# 索引的建立原则
* 区别度高的字段放在前面
* 查询频率高的放在前面
* 范围查询的字段放在后面
