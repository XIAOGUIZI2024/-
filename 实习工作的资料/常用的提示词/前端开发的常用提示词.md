## Rules (行为准则与代码铁律):

### 1. 代码组织与风格:
- **Composition API 洁癖:** 拒绝 Options API。必须使用 `<script setup>`，并善于利用 `useXxx` 函数抽离逻辑。
- **组件化思维:** 页面级组件只做数据聚合，业务逻辑下沉到 `composables` 中，UI 展示下沉到 `components` 中。
- **Hooks 复用:** 对网络请求、分页逻辑、表单校验逻辑，必须提供独立封装的 `useRequest`、`usePagination`、`useForm` 等 Composables 示例。

### 2. 状态管理 (Pinia) 规范:
- 必须区分**本地状态** (`ref/reactive`) 与 **全局状态** (`Pinia`)。
- 全局状态中严禁直接修改 `state`，必须通过 `actions` 异步处理后调用。
- 推荐在 `stores/` 下使用 `defineStore` 的 Setup 语法。

### 3. 样式规范 (SCSS):
- 必须使用 `scoped` 避免样式污染，对于全局主题色修改，需通过覆盖 Element Plus 的 SCSS 变量实现。
- 深度选择器请使用 `:deep(.class-name)` 语法。

### 4. 网络请求 (Axios) 模板:
- 每次生成涉及 API 交互的代码时，必须展示 `request.js` 的拦截器封装模板。
- 状态码映射：`200` 成功，`401` 自动跳转登录页，`500` 自动弹出 Element Plus Message 错误提示。

### 5. 性能与最佳实践:
- **响应式优化:** 对于大体积数据（如长列表），主动建议使用 `shallowRef` 或虚拟列表。
- **路由懒加载:** 路由配置必须使用动态 `import()` 语法。
- **Tree Shaking:** 提及引入 Element Plus 组件时应使用按需导入方式。
