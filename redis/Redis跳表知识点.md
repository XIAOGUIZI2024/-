## 一、跳表（SkipList）核心基础（解答“跳表是什么”“实现思路”）

### 1. 跳表的本质

跳表是**有序链表的升级版**，核心目的是解决普通有序链表查找、插入、删除效率低（O(n)）的问题，通过“多层索引+底层全量有序链表”，将时间复杂度优化至 O(log n)，主要用于 Redis 有序集合（ZSet）的底层实现。

### 2. 核心实现思路

核心逻辑：底层全量有序链表 + 上层多层稀疏索引 + 随机生成层数 = 能跳跃的有序链表，具体拆解如下：

- 底层（Level 0）：存储全部真实数据，是完整的有序链表，数据按 score 升序排列（score 相同则按 member 字典序）；
    
- 上层（Level 1 及以上）：稀疏索引层，层数越高，节点越少、跨度越大，用于快速跳跃定位，减少无效遍历；
    
- 节点结构：每个节点包含 ele（member）、score、多层指针（level[]）、后退指针，多层指针是实现“跳跃”的核心；
    
- 核心操作逻辑：查找（从最高层往下，能右则右，不能右则降层）、插入（定位位置→随机层高→逐层串入链表）、删除（定位节点→逐层断链→回收节点）。
    

## 二、跳表分层分布（重点解答“数据是否会放在不同层”“分层规则”）

### 1. 分层核心规则（Redis 标准）

跳表的分层 **不是人为划分**，而是每个新节点通过“随机概率”决定自身层数，规则如下：

- 所有节点最少 1 层（必在 Level 0 底层）；
    
- 每往上晋升一层的概率为 25%（1/4）；
    
- 最高层数限制为 32 层，防止层数爆炸。
    

### 2. 关键疑问解答：一个数据会出现在不同层吗？

会，但 **数据本体只存 1 份**，不会重复存储：

- 一个节点如果随机到 3 层（如示例中的 50），就会同时挂在 Level 0、Level 1、Level 2 三层链表中；
    
- 本质是同一个节点，只是自带多层指针，上层链表通过指针引用该节点，而非复制数据；
    
- 层数越多的节点，越适合作为高层索引，实现大步跳跃，提升查询效率（如示例中 50 作为高层索引，快速定位 90）。
    

### 3. 分层分布规律

天然呈现“上少下多、上层稀疏、下层密集”的特点：Level 0 包含所有节点，Level 1 约 1/4 节点，Level 2 约 1/16 节点，层数越高，节点越少、跨度越大。

## 三、跳表的删除开销（解答“删除数据开销是否很大”）

结论：**开销不大，完全可控，时间复杂度稳定 O(log n)**，原因如下：

- 层数有限：绝大多数节点只有 1~2 层，最高 32 层封顶，即使逐层删除，也只是固定次数的指针修改，开销可忽略；
    
- 操作简单：删除前查找节点时，会同步记录每层的前驱节点，找到目标后，只需逐层修改前驱节点的指针，跳过目标节点，无需复杂计算；
    
- 对比优势：相比红黑树的旋转变色、递归平衡，跳表删除无需复杂平衡操作，实际开销更低；
    
- 局部影响：删除仅影响目标节点所在的几层，不引发全局结构调整，进一步降低开销。
    

## 四、跳表的缺点（补充完善，贴合疑问延伸）

- 内存开销高：每个节点需维护多层指针、跨度等信息，平均自带 3~4 层指针，比普通链表内存冗余更大（用内存换速度）；
    
- 写操作略弱：插入、删除需逐层修改指针，步骤比红黑树多，写性能略逊于红黑树；
    
- 非严格平衡：依赖随机算法，极端情况下（概率极低）会退化为普通链表，时间复杂度变为 O(n)（Redis 通过 32 层上限压低成本）；
    
- 缓存不友好：链表节点内存碎片化、不连续，高层跳跃访问时，CPU 缓存命中率较低。
    

## 五、跳表的实际应用场景（重点突出 Redis ZSet）

跳表的核心应用场景是 Redis ZSet，此外还有以下场景：

- Redis ZSet（最经典）：底层核心实现，支撑排行榜、范围查询、排名等功能；
    
- 存储引擎：LevelDB、RocksDB 的 MemTable 底层用跳表，适配 LSM 树架构，支持海量数据有序写入；
    
- Java 并发容器：ConcurrentSkipListMap、ConcurrentSkipListSet，适合高并发场景下的有序操作；
    
- 时序数据库、搜索引擎：适合有序数据的范围查询、时间线检索等场景。
    

## 六、Redis ZSet 底层协同：跳表 + 哈希字典（重点解答“是否只靠跳表”“共享数据”“协同工作”）

### 1. 核心结论：ZSet 不是只靠跳表，而是“跳表 + 哈希字典”协同工作

两者缺一不可，互补优缺点，共同支撑 ZSet 的所有功能，且 **共享同一份数据，不重复存储**。

### 2. 各自职责

- 哈希字典（dict）：key = member（成员），value = 数据节点指针，负责 O(1) 快速单点查询（如 ZSCORE 命令）、判断成员是否存在、快速定位节点；
    
- 跳表（skiplist）：通过指针串联所有数据节点，按 score 有序排列，负责范围查询（如 ZRANGE）、排名（如 ZRANK）、有序遍历。
    

### 3. 关键疑问解答：“共享同一份数据”是什么意思？

全局只有 1 份真实数据节点（zskiplistNode），存储 ele（member）和 score，跳表和哈希字典均通过指针引用该节点，不复制数据：

- 跳表：直接串联节点指针，维护有序结构；
    
- 哈希字典：仅存储“member → 节点指针”的映射，不存储 score 和 member 本身；
    
- 修改数据时，只需修改唯一的节点，跳表和字典通过指针同步感知，无需分别修改。
    

### 4. 两者协同工作流程（核心操作）

- 新增（ZADD）：先查字典判重 → 创建唯一节点 → 插入跳表（维持有序） → 字典存入“member→节点指针”；
    
- 查询分数（ZSCORE）：纯走字典，O(1) 拿到节点指针，读取 score，不碰跳表；
    
- 范围查询（ZRANGE）：纯走跳表，遍历有序节点，读取数据，不依赖字典；
    
- 修改分数（ZADD 覆盖）：字典快速定位节点 → 修改 score → 跳表中删除该节点 → 按新 score 重新插入跳表（维持有序）；
    
- 删除（ZREM）：字典 O(1) 定位节点 → 跳表中逐层删除节点 → 字典删除“member→节点指针” → 释放节点内存。
    

### 5. 补充例外

当 ZSet 元素数量少、元素值短时，Redis 会用 ziplist（压缩列表）作为底层，节省内存；当数据量增大，自动切换为“跳表 + 哈希字典”。

## 七、全程核心疑问汇总（快速回顾）

1. 跳表是什么？—— 有序链表的升级版，通过多层索引优化效率，用于 Redis ZSet 底层；
    
2. 跳表如何分层？—— 节点随机决定层数（25% 晋升概率，最高 32 层），上层稀疏、下层密集；
    
3. 一个数据会放在不同层吗？—— 会，但数据只存 1 份，不同层共享同一个节点的指针；
    
4. 删除数据开销大吗？—— 不大，最多修改 32 层指针，操作简单，无复杂平衡；
    
5. ZSet 只靠跳表吗？—— 不是，是跳表 + 哈希字典协同，各司其职；
    
6. “共享同一份数据”是什么意思？—— 只有 1 份真实节点，跳表和字典均通过指针引用，不重复存储。
