## 1. 基础加锁 & 解锁

- 加锁用：`SET key 唯一值 NX PX 过期时间`，保证**原子性 + 互斥 + 防死锁**
- 解锁**不能直接 DEL**，必须用 **Lua 脚本**：先判断 value 是否是自己的，是才删
- 核心原则：**谁加的锁，谁才能删**，不是自己的锁绝不碰

## 2. 任务执行中锁过期的问题

- 业务执行时间 > 锁过期时间 → 锁自动失效 → 别的线程能抢到锁 → **并发安全被破坏**
- 手写 Redis 锁**无法解决**，只能靠**锁续期（看门狗）**

## 3. 看门狗机制（核心重点）

- 看门狗 = 后台定时线程，**每隔一段时间自动给锁续期**
- 默认：锁 30s 过期，每 10s 续一次，重置为 30s
- 续期命令：`PEXPIRE`，配合 Lua 保证**只续自己的锁**
- 服务挂掉 → 看门狗一起挂 → 锁自动过期，**不会死锁**

## 4. 你发现的极限 Bug

- 在**续期临界点**（比如第 10s、第 20s），如果网络抖动 / GC/Redis 延迟
- 可能出现：**锁先过期 → 别人加锁 → 续期命令才到**
- Redis 锁**无法 100% 避免这个时间窗口**，只能降低风险
- 这是 Redis 分布式锁**天生缺陷**，不是代码问题

## 5. Redis 集群下的锁分配

- **普通 Redis Cluster**：
    
    锁 key 通过哈希槽路由到**单个 master 节点**
    
    优点：快、简单
    
    缺点：主从切换可能丢锁
    
- **Redlock 红锁**：
    
    向多个独立 master 加锁，**多数节点成功才算加锁成功**
    
    解决主从脑裂问题，但更重、更慢
    

## 6. 最终结论

- Redis 分布式锁 = **够用、高性能，但不是绝对强一致**
- 能防大部分并发问题，但**存在极小概率的锁失效时间窗口**
- 追求绝对安全 → 用 Zookeeper / Etcd
- 追求性能 + 大部分场景安全 → 用 Redis + Redisson（看门狗）
