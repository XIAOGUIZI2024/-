1. **热门 Key 概念与危害**
    
    热门 Key 是短时间内超高 QPS 集中访问的同一个 Key，会造成单节点 CPU 打满、流量倾斜、缓存击穿、服务雪崩等问题。
    
2. **单机 Redis 处理热 Key 能力**
    

- 单机**无法通过拆分 Key、多节点分流**的方式解决热 Key（单线程执行、无节点分摊）；
- 只能靠**业务层方案缓解**：应用本地缓存、随机过期、接口限流、逻辑过期等，满足中小流量业务；
- 高并发秒杀、大流量 C 端场景，单机 Redis 扛不住。

3. **集群 Redis 处理热 Key 优势**

- 可通过**Key 分片、读写分离、多从库分担读请求**，将流量打散到多个节点；
- 是高并发场景下，**根治热 Key**的核心方案。

4. **冷热 Key 识别方案（重点）**

- 临时排查：`redis-cli --hotkeys`（依赖 LFU），禁止长期使用`MONITOR`；
- 线上常规方案：**客户端埋点统计 Key 访问频次**（通用、无侵入）；
- 架构层方案：Redis 代理 / 网关、云数据库自带 TopKey 监控；
- 可视化：Prometheus+Grafana、日志聚合分析辅助发现。

5. **核心结论**

- 依靠 Redis 自身分布式能力分流压力，**必须使用集群**；
- 小项目、低并发场景，**单机 Redis + 本地缓存**即可解决热 Key 问题；
- 热 Key 标准处理组合：**发现监控 + 流量拦截（本地缓存）+ 分片分流 + 限流降级兜底**。
