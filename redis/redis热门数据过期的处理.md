## 一、Redis 三种过期删除策略

1. **惰性删除（被动）**
    
    访问 key 时才检查，**过期就删、返回空**；没过期正常返回。
    
    优点：省 CPU；缺点：冷门过期 key 一直占内存。
    
2. **定期删除（主动）**
    
    每 100ms 随机抽查部分过期 key 清理，**不全部遍历**。
    
    原因：Redis 单线程，全量删除会阻塞主线程、压垮性能；抽样渐进清理，平衡 CPU 与内存。
    
3. **内存淘汰（兜底策略）**
    
    内存打满时触发，共 8 种：
    

- `volatile-xxx`：只删**设置过过期**的 key
- `allkeys-xxx`：删除**所有**key，不分过期
- `noeviction`：不删数据，写入直接报错

---

## 二、LRU 与 LFU 核心区别

- **LRU（R=Recently 最近）**：看**时间**，淘汰**最久没使用**的数据
- **LFU（F=Frequently 频率）**：看**次数**，淘汰**使用频率最低**的数据

---

## 三、主从复制：从节点同步到过期数据 问题 + 解决

1. 核心规则
    
    **主节点**：负责过期判断、惰性 / 定期删除；
    
    **从节点**：不会主动删除过期 key，只等待主节点同步 `DEL` 命令。
    
2. 问题
    
    主库 key 已过期但未删除，从库会保留过期数据，可能读到脏数据。
    
3. 解决方案
    
4. 高版本 Redis (3.2+)：从库**读取时校验 TTL**，过期直接返回空，不返回脏数据；
    
5. 主库删除过期 key 后，同步 `DEL` 命令，从库最终物理删除；
    
6. 配置 `replica-serve-stale-data no` 禁止从库提供过期旧数据；
    
7. 业务兜底：核心强一致业务**读主库**。
    

---

## 四、整体串联（面试一句话完整版）

Redis 通过**惰性删除 + 定期删除**清理过期数据，内存溢出时触发**内存淘汰**兜底；

LRU 按最近使用时间淘汰，LFU 按访问频率淘汰；

主从架构下由主库统一管理过期删除，从库不主动清理，高版本从库做逻辑过期拦截，配合主库删除指令同步与业务读写分离，解决从库过期数据问题。
