## 1. Redis 事务基础

1. Redis 支持事务（`MULTI/EXEC/DISCARD`），**但不是完整 ACID 事务**。
2. 最大特点：**无回滚**，命令执行报错，已执行的不会撤销。
3. 为什么无回滚：Redis 设计理念追求**极简、高性能、轻量**；回滚需要日志、缓存旧数据，增加开销；且运行时错误大多是代码问题，不该靠数据库兜底。

## 2. 完整事务

完整事务 = 严格满足 **ACID**：

- 原子性：要么全成功，要么全回滚
- 一致性：数据约束、业务规则合法
- 隔离性：并发互相隔离
- 持久性：提交后数据永久落库
    
    代表：MySQL InnoDB；Redis 原生事务不满足。

## 3. Redis 事务 VS 乐观锁（WATCH）

1. **Redis 事务（MULTI）**
    
    作用：多条命令入队，`EXEC` 时**批量连续执行**，防止命令插队；
    
    缺陷：管不了「别人提前修改数据」，解决不了并发竞争。
2. **WATCH 乐观锁**
    
    作用：监视**一个 / 多个 key**，事务提交前，只要被监听 key 被其他客户端修改，事务直接失效；
    
    必须搭配事务使用，本质是**版本校验**，无阻塞、无加锁。
3. 关系：事务保证**执行不中断**，WATCH 保证**并发数据安全**，二者搭配使用。

## 4. 核心疑惑：Redis 单线程，为什么还会并发超卖？

1. Redis 是**单线程命令串行**：同一时刻只执行一条命令。
2. 关键误区：
    
    单线程只保证「**单条命令**不并发」，**不保证一个客户端多条业务命令连续独占**。
3. 超卖根源：
    
    业务逻辑是 **多条命令 + 本地代码判断**（查库存→本地判断→扣库存），
    
    两条命令中间会释放执行权，其他客户端可以插队查询，拿到旧数据，最终并发覆盖、超卖。

## 5. Lua 脚本 解决并发问题

1. 把 **查询 + 判断 + 修改** 全部写在一个 Lua 脚本中。
2. Redis 会把整个 Lua 脚本当做**一条不可分割的执行单元**。
3. 执行特性：脚本从头到尾跑完，**中途不切换客户端、不处理其他请求**，天然串行、不插队、原子执行，彻底解决超卖。

## 6. 隔离级别核心

1. Redis 没有 MySQL 那种多种隔离级别配置。
2. 只有两种场景天然实现 **串行化隔离级别**：
    
    - `EXEC` 执行事务队列的**瞬间**，批量命令连续执行，不被打断；
    - **Lua 脚本全程执行期间**，独占单线程，完全串行隔离。
    
3. 零散普通命令无隔离，会互相穿插，产生并发问题。

---

### 终极串联（一句话串全流程）

Redis 原生事务只有批量执行能力、无回滚，依靠单线程只能保证单条命令串行；多步业务操作会被插队引发并发问题，所以用 WATCH 做乐观锁冲突检测，或用 Lua 脚本 / 事务批量执行的串行化隔离特性，实现并发数据安全。
